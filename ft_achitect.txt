5. Memory / Ownership Diagram
The evaluator may ask: "Who owns what? Who deletes what?"
plain

┌─────────────────────────────────────────────────────────────┐
│                        OWNERSHIP MAP                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  [Server] owns:                                              │
│    ├── _clients (map) ──▶ Client objects (value)             │
│    ├── All_chanel (map) ──▶ chanel objects (value)          │
│    └── exCmd (map) ──▶ execute* pointers (HEAP)             │
│                                                              │
│  [Client] owns:                                              │
│    ├── Its own std::string buffers (recv/send)               │
│    └── chanel_client map ──▶ POINTERS to chanel             │
│        (these are NON-owning references)                     │
│                                                              │
│  [chanel] owns:                                              │
│    ├── all_Cchanel map ──▶ POINTERS to Client               │
│    ├── op_chanel map ──▶ POINTERS to Client                 │
│    └── wait_list set ──▶ std::string nicknames              │
│                                                              │
│  [execute*] are owned by Server, deleted in ~Server()       │
│                                                              │
└─────────────────────────────────────────────────────────────┘

Key defense point: "Clients hold pointers to channels, and channels hold pointers to clients. Neither deletes the other. The Server deletes the actual objects. When a client disconnects, removeClient() erases the pointer from the channel's maps first, then erases the client from _clients."

8. Quick Reference: File Responsibilities
Table
File	Responsibility	Evaluator Question
main.cpp	Validates args, constructs Server, calls run()	"How does the server start?"
server.cpp	Socket I/O, epoll loop, client lifecycle, containers	"How do you handle 100 clients?"
client.cpp	Data model for a user: nick, buffers, flags, channel list	"What does a client object store?"
channel.cpp	Data model for a room: members, ops, topic, modes	"How do you know who is an operator?"
execute.cpp	IRC protocol logic, replies, error numerics	"What happens on JOIN with a wrong password?"
Final tip: If the evaluator asks you to draw this on a whiteboard, redraw View 1 (System Architecture) and View 4 (State Machine). Those two diagrams alone prove you understand the concurrency model and the protocol requirements. Everything else is supporting detail.
